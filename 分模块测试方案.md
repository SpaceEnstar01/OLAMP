# 分模块测试方案设计

## 一、需求分析

### 目标
1. 单独测试LCD屏幕模块
2. 单独测试语音模块（音频编解码）
3. 单独测试Camera摄像头模块
4. 不破坏原有代码的核心逻辑
5. 可以独立编译和运行测试程序

### 约束条件
- 保持原有 `main/` 目录下的代码不变
- 复用现有的Board类和硬件初始化代码
- 测试代码与主程序代码隔离

---

## 二、方案设计

### 方案A：独立测试目录 + 条件编译（推荐）

#### 2.1 目录结构
```
xiaozhi-esp32/
├── main/
│   ├── ... (原有代码保持不变)
│   └── TEST/                    # 新建测试目录
│       ├── camera_test/
│       │   ├── CMakeLists.txt
│       │   ├── main.cc
│       │   └── camera_test.h
│       ├── voice_test/
│       │   ├── CMakeLists.txt
│       │   ├── main.cc
│       │   └── voice_test.h
│       └── screen_test/
│           ├── CMakeLists.txt
│           ├── main.cc
│           └── screen_test.h
├── CMakeLists.txt              # 根CMakeLists需要修改
└── ...
```

#### 2.2 核心思路

**1. 测试程序入口点**
- 每个测试程序有自己的 `app_main()` 函数
- 通过CMake配置选择编译主程序还是测试程序
- 使用 `idf_component_register` 的 `REQUIRES` 来复用现有组件

**2. Board类的使用**
- 测试程序直接调用 `Board::GetInstance()` 获取硬件接口
- 由于Board构造函数会初始化所有模块，但测试程序只使用需要的模块
- 对于不需要的模块，可以获取后不调用其方法

**3. 编译隔离**
- 在根 `CMakeLists.txt` 中添加测试目标
- 使用 `idf_build_get_property` 和条件判断来选择编译目标
- 或者使用独立的 `sdkconfig.test.*` 配置文件

#### 2.3 实现细节

**方案A-1：通过CMake变量控制编译目标**

```cmake
# 根 CMakeLists.txt 中添加
if(TEST_TARGET)
    # 编译测试程序
    set(MAIN_COMPONENT "test_${TEST_TARGET}")
else()
    # 编译主程序（默认）
    set(MAIN_COMPONENT "main")
endif()
```

**方案A-2：通过独立的测试组件**

每个测试目录作为一个独立的ESP-IDF组件：
- 测试组件可以 `REQUIRES main` 来访问主程序的代码
- 但需要解决 `app_main()` 冲突问题（可以通过链接顺序或条件编译解决）

**方案A-3：修改main组件支持测试模式（推荐）**

在 `main/CMakeLists.txt` 中添加测试模式：
```cmake
# 如果定义了测试目标，则包含测试代码
if(CONFIG_TEST_TARGET)
    # 包含测试目录的源文件
    file(GLOB TEST_SOURCES "TEST/${CONFIG_TEST_TARGET}/*.cc")
    list(APPEND SOURCES ${TEST_SOURCES})
    # 使用测试的main.cc替代原来的main.cc
    list(REMOVE_ITEM SOURCES "main.cc")
    list(APPEND SOURCES "TEST/${CONFIG_TEST_TARGET}/main.cc")
endif()
```

### 方案B：独立项目目录（更彻底隔离）

#### 2.1 目录结构
```
xiaozhi-esp32/
├── main/                        # 原有代码
├── test_projects/               # 新建测试项目目录
│   ├── camera_test/
│   │   ├── CMakeLists.txt
│   │   ├── main/
│   │   │   ├── CMakeLists.txt
│   │   │   └── main.cc
│   │   └── sdkconfig
│   ├── voice_test/
│   │   └── ...
│   └── screen_test/
│       └── ...
└── CMakeLists.txt
```

#### 2.2 核心思路
- 每个测试作为一个独立的ESP-IDF项目
- 通过符号链接或CMake的 `add_subdirectory` 来复用主程序的代码
- 完全隔离，但需要维护多个项目配置

---

### 方案C：运行时选择模式（最简单但不够灵活）

#### 2.1 实现方式
在原有的 `main.cc` 中添加测试模式选择：
```cpp
extern "C" void app_main(void) {
    // 通过NVS或GPIO来选择运行模式
    bool test_mode = GetTestMode();
    if (test_mode) {
        RunTestModule();
    } else {
        // 原有逻辑
        Application::GetInstance().Start();
    }
}
```

**缺点**：需要修改原有代码，不够优雅

## 三、推荐方案：方案A-3（修改main组件支持测试模式）

### 3.1 优势
1. ✅ **最小侵入**：只需要修改 `main/CMakeLists.txt`，不改变原有代码逻辑
2. ✅ **复用性好**：测试程序可以直接使用Board类和所有硬件初始化代码
3. ✅ **编译灵活**：通过menuconfig或CMake变量选择编译目标
4. ✅ **维护简单**：测试代码集中在一个目录，易于管理

### 3.2 实现步骤

#### 步骤1：创建测试目录结构
```
main/TEST/
├── camera_test/
│   ├── CMakeLists.txt
│   └── main.cc
├── voice_test/
│   ├── CMakeLists.txt
│   └── main.cc
└── screen_test/
    ├── CMakeLists.txt
    └── main.cc
```

#### 步骤2：修改 `main/CMakeLists.txt`
在文件末尾添加：
```cmake
# 测试模式支持
if(CONFIG_TEST_TARGET)
    set(TEST_TARGET ${CONFIG_TEST_TARGET})
    message(STATUS "Building test target: ${TEST_TARGET}")
    
    # 移除原有的main.cc
    list(REMOVE_ITEM SOURCES "main.cc")
    
    # 添加测试程序的main.cc
    list(APPEND SOURCES "TEST/${TEST_TARGET}/main.cc")
    
    # 添加测试目录到包含路径
    list(APPEND INCLUDE_DIRS "TEST/${TEST_TARGET}")
endif()
```

#### 步骤3：添加Kconfig选项
在 `main/Kconfig.projbuild` 中添加：
```
menu "Test Configuration"
    config TEST_TARGET
        string "Test target name"
        default ""
        help
            Set to "camera_test", "voice_test", or "screen_test" to build test program.
            Leave empty to build normal application.
endmenu
```

#### 步骤4：编写测试程序模板

**screen_test/main.cc 示例结构：**
```cpp
#include "board.h"
#include "display.h"
#include <esp_log.h>

extern "C" void app_main(void) {
    ESP_LOGI("SCREEN_TEST", "Starting screen test...");
    
    // 初始化基本系统（NVS等）
    // ...
    
    // 获取Board实例（会自动初始化所有硬件）
    auto& board = Board::GetInstance();
    
    // 获取Display实例
    auto* display = board.GetDisplay();
    if (display == nullptr) {
        ESP_LOGE("SCREEN_TEST", "Display is null!");
        return;
    }
    
    // 执行屏幕测试
    // - 显示测试图案
    // - 测试颜色
    // - 测试刷新率
    // - 测试触摸（如果有）
    
    ESP_LOGI("SCREEN_TEST", "Screen test completed");
}
```

**voice_test/main.cc 示例结构：**
```cpp
#include "board.h"
#include "audio_codec.h"
#include <esp_log.h>

extern "C" void app_main(void) {
    ESP_LOGI("VOICE_TEST", "Starting voice test...");
    
    // 初始化基本系统
    // ...
    
    // 获取Board实例
    auto& board = Board::GetInstance();
    
    // 获取AudioCodec实例
    auto* codec = board.GetAudioCodec();
    if (codec == nullptr) {
        ESP_LOGE("VOICE_TEST", "AudioCodec is null!");
        return;
    }
    
    // 启动编解码器
    codec->Start();
    
    // 执行语音测试
    // - 测试录音
    // - 测试播放
    // - 测试音量控制
    // - 测试回声消除（如果有）
    
    ESP_LOGI("VOICE_TEST", "Voice test completed");
}
```

**camera_test/main.cc 示例结构：**
```cpp
#include "board.h"
#include "camera.h"
#include <esp_log.h>

extern "C" void app_main(void) {
    ESP_LOGI("CAMERA_TEST", "Starting camera test...");
    
    // 初始化基本系统
    // ...
    
    // 获取Board实例
    auto& board = Board::GetInstance();
    
    // 获取Camera实例
    auto* camera = board.GetCamera();
    if (camera == nullptr) {
        ESP_LOGE("CAMERA_TEST", "Camera is null!");
        return;
    }
    
    // 执行摄像头测试
    // - 测试图像捕获
    // - 测试不同分辨率
    // - 测试图像质量
    // - 测试帧率
    
    ESP_LOGI("CAMERA_TEST", "Camera test completed");
}
```

### 3.3 编译和使用

#### 编译测试程序
```bash
# 方法1：通过menuconfig设置
idf.py menuconfig
# 在 "Test Configuration" -> "Test target name" 中设置测试目标

# 方法2：通过CMake变量（如果支持）
idf.py -DTEST_TARGET=camera_test build

# 方法3：创建测试专用的sdkconfig
cp sdkconfig sdkconfig.test.camera
# 在sdkconfig.test.camera中设置 CONFIG_TEST_TARGET="camera_test"
idf.py -s sdkconfig.test.camera build
```

#### 运行测试
```bash
idf.py flash monitor
```

## 五、注意事项

### 5.1 Board初始化问题
- **问题**：Board构造函数会初始化所有模块（Display、AudioCodec、Camera等）
- **影响**：即使测试程序只使用一个模块，其他模块也会被初始化
- **解决方案**：
  - 方案1：接受这个行为，测试时只关注目标模块
  - 方案2：修改Board类支持延迟初始化（需要改动原有代码，不推荐）
  - 方案3：为测试创建轻量级Board子类（需要改动原有代码，不推荐）

### 5.2 依赖问题
- **问题**：测试程序可能需要一些主程序的依赖（如NVS、事件循环等）
- **解决**：在测试程序的main.cc中初始化必要的系统组件

### 5.3 资源冲突
- **问题**：如果Board初始化了所有模块，可能存在资源冲突
- **解决**：确保测试程序只使用目标模块，其他模块获取后不使用

### 5.4 编译配置
- **问题**：不同测试可能需要不同的sdkconfig配置
- **解决**：为每个测试创建独立的sdkconfig文件（如 `sdkconfig.test.camera`）

## 八、总结

**推荐使用方案A-3**，因为：
1. ✅ 对原有代码侵入最小
2. ✅ 可以完全复用现有的Board类和硬件初始化代码
3. ✅ 测试代码集中管理，易于维护
4. ✅ 编译灵活，可以通过配置选择测试目标
5. ✅ 不影响主程序的正常开发

这个方案可以让你在不破坏原有代码逻辑的前提下，独立开发和测试各个硬件模块。
